프로세스에 입각한 바이브코딩의 효과 분석
Lessons Learned

1. 요구분석 단계 (Chap 04, 05)

1-1. 효과적이었던 점

Chap 04 요구분석 방법론을 프롬프트에 명시하자
AI가 기능 요구사항과 비기능 요구사항을 구분하여 명세서를 작성하였다.
요구 품질 기준인 원자성, 완전성, 비모호성이 대체로 충족되었다.
유스케이스 포함 관계와 확장 관계를 명확히 정의할 수 있었다.

1-2. 한계점

AI가 생성한 요구사항에는 이해관계자의 실제 업무 맥락이 반영되지 않았다.
도메인 지식과 실제 업무 경험은 AI가 대체할 수 없었다.
요구사항 도출 과정에서 실제 경험 기반의 아이디어는 개발자가 직접 제공해야 하였다.


2. 설계 단계 (Chap 06, 07)

2-1. 효과적이었던 점

MVC 패턴을 설계 단계에서 명확히 정의하고 구현하자
기능 추가 시 영향 범위가 명확하였다.
예를 들어 필터 기능 추가 시 Model의 filter() 메서드만 수정하면 되었다.
SOLID 원칙을 프롬프트에 명시하자 AI가 원칙에 맞는 구조로 코드를 생성하였다.
OCP 원칙 적용으로 초기 3개 필터에서 7개로 확장할 때 기존 코드 수정 없이 가능하였다.

2-2. 한계점

AI는 SOLID 원칙을 프롬프트에 명시해야 적용하였다.
명시하지 않으면 동작하는 코드를 생성하지만 유지보수성이 낮은 코드를 생성하였다.
이는 소프트웨어 공학 지식이 바이브코딩의 품질을 결정한다는 것을 보여준다.

2-3. Polyglot Architecture 관점

현재 단일 언어(JavaScript)로 구현된 단순 구조이다.
단일 사용자 로컬 환경이므로 JavaScript 단일 언어 구조가 적합한 선택이었다.
향후 멀티 사용자 협업 기능 확장 시에는
Node.js(서버), JavaScript(클라이언트), SQL(데이터베이스)의
Polyglot 구조가 필요할 것이다.
분산 병렬 환경에서는 Erlang 계열 언어 도입 검토가 필요하다.


3. 구현 단계 (Chap 09)

3-1. 효과적이었던 점

코딩 표준을 프롬프트에 명시하자 AI가 일관된 명명 규칙과 주석 구조를 유지하였다.
카멜케이스 명명 규칙이 일관되게 적용되었다. (todoInput, dueDateInput, renderList 등)
섹션별 구조 주석으로 코드 가독성이 향상되었다.
리팩토링 요청 시 AI가 코드 스멜을 발견하고 개선하였다.
중복 코드 제거: 우선순위 탭 선택 로직을 selectPriority()와 selectEditPriority()로 분리
긴 메서드 분리: renderList()에서 renderStats()를 별도 메서드로 분리

3-2. 한계점

AI가 생성한 코드에도 인스펙션이 필요하였다.
설계 없이 기능을 추가하자 toggleDark() 함수가 TodoController 객체 외부에
정의되는 구조적 오류가 발생하였다.
이는 아키텍처 설계를 먼저 확정하고 구현했다면 발생하지 않았을 문제이다.
요구사항 분석 없이 기능을 추가하자 스코프 크리프(Scope Creep)가 발생하였다.


4. 품질관리 단계 (Chap 12)

4-1. 정적 분석 (Static Analysis)

인스펙션은 행위(실행) 기반이 아닌 정적 분석 방식으로 수행하였다.
코드를 눈으로 보고 실행 경로를 정적으로 분석하여 코드 안전성을 증명하였다.

사이클로매틱 복잡도 분석 결과
- filter() 함수: 분기 5개, V(G) = 6, 허용 범위 내
- addTodo() 함수: 분기 4개, V(G) = 5, 적정 수준
- isOverdue() 함수: 분기 1개, V(G) = 2, 단순하고 안전

코드 안전성 증명
- isOverdue(): dueDate null 체크 후 false 반환 -> null 참조 오류 없음
- filter(): 알 수 없는 type 입력 시 전체 목록 반환 -> 예외 상황 안전
- toggle(), update(): id 미발견 시 조기 반환 -> null 참조 오류 없음

4-2. 테스트 케이스 기반 검증

결함 수정 시 전체 코드를 재검토하지 않고
테스트 조건을 먼저 설정하고 범위를 한정하여 검증하였다.

iOS 과거 날짜 선택 방지 결함 수정 사례
- 결함 범위 설정: addTodo() 함수의 날짜 유효성 검사 부분만 한정
- 테스트 조건: 오늘 날짜 선택, 미래 날짜 선택, 과거 날짜 선택 3가지
- 검증 결과: min 속성이 iOS에서 무시됨을 확인
- 해결: JS 유효성 검사로 방어적 코딩 적용

이 방식은 전체 코드를 재검토하는 것보다 효율적이었으며
다른 기능에 영향을 주지 않았다.

4-3. 인스펙션 결과 요약

SOLID 원칙 5개 항목 전체 준수
UI 사용성 원칙 7개 항목 전체 적용
코딩 표준 5개 항목 전체 준수


5. 종합 결론

5-1. 프로세스를 따랐을 때 효과적이었던 사례

MVC 패턴 사전 설계 -> 기능 추가 시 영향 범위 명확
코딩 표준 명시 -> AI가 일관된 코드 생성
정적 분석 기반 인스펙션 -> 실행 없이 잠재적 결함 사전 발견
테스트 범위 한정 -> 효율적인 결함 수정

5-2. 프로세스를 따르지 않았을 때의 문제

설계 없이 기능 추가 -> 구조적 오류 발생 (toggleDark 위치 오류)
요구분석 없이 기능 추가 -> 스코프 크리프 발생
코딩 표준 미명시 -> 유지보수성 낮은 코드 생성

5-3. 최종 결론

소프트웨어 공학 프로세스를 바이브코딩에 적용한 결과
단순히 AI가 생성한 코드를 사용하는 것보다
품질과 유지보수성이 높은 결과물을 얻을 수 있었다.

특히 정적 분석 기반 인스펙션과 테스트 케이스 기반 검증은
AI 생성 코드의 신뢰성을 높이는 데 효과적이었다.

바이브코딩은 개발 속도를 높이는 도구이지만
소프트웨어 공학 지식이 그 품질을 결정한다는 것을 확인하였다.
